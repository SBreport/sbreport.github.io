## 1. 프로젝트 개요

### 1.1 무엇을
네이버 키워드를 분석하는 **로그인 기반 웹앱**. 키워드를 입력하면 해당 키워드의 검색 결과 구좌 구성(어떤 영역이 어떤 순서로 노출되는지)과 검색량을 분석해 보여준다.

### 1.2 왜
- 기존 Chrome 확장(스마트 브랜딩 서포터, SBS)은 사용자가 네이버 검색 페이지에 있을 때만 동작.
- 웹앱은 키워드만 입력하면 분석 결과를 한눈에 볼 수 있는 독립 서비스.
- 외부 클라이언트 대상 + 추후 유료 전환 예정.

### 1.3 SBS 확장과의 관계
- **독립 운영**. 확장 ↔ 웹앱 자동 연동 없음 (사용자 결정).
- 확장은 확장대로(네이버 페이지 위에서), 웹앱은 웹앱대로(키워드 입력 분석).
- Cloudflare Worker(`naver-searchad-proxy`)는 공유 자산.

### 1.4 핵심 목표 3가지 (사용자 명시)
1. 로그인 시스템 (구글 인증, 다중 사용자, 외부 클라이언트 포함)
2. 확장에서 구현한 기능을 웹에서 분석·조회
3. 추후 로그인 이용자 대상 유료 전환 (보안 토대 미리 준비)


## 4. 디자인 시스템 룰 (사시모드 방지)

코드·토큰·린트로 강제 가능한 것만 채택. "지키자"는 다짐 금지.

### 4.1 컬러
- primary = 브랜드 블루 `#1684F2` (Nuxt UI primary 커스터마이즈), gray = slate
- 액센트는 primary 1색. 상태색(red/green/amber)은 상태 표시에만. 표·본문은 무채색 90%.

### 4.2 텍스트
- 기본 본문/표 셀 `text-sm` (14px). 메타 `text-xs` (12px)이 최소. 11px 이하 금지.
- 폰트 Pretendard, 줄간격 1.5.

### 4.3 여백 (Tailwind 4px 기반, **32px가 상한**)
- 기본 컴포넌트 내부 `p-3`(12px), 카드 `p-4`, 섹션 분리 `gap-6`. `gap-8`(32px) 초과 금지.

### 4.4 레이아웃 — 사시모드 박멸 핵심
- 컨테이너 `mx-auto` 중앙정렬 **금지** (사시모드 1순위 원인). 본문 좌측 정렬 시작.
- 데이터 영역 max-width 상한 없음. 사이드바 240px 고정. 헤더 56px.

### 4.5 전체 스크롤 박멸 + 헤더/사이드바 잘림 방지 (사용자 강조)
- `html, body { height:100%; overflow:hidden }` — body 절대 스크롤 안 함.
- page root `h-screen flex flex-col overflow-hidden` (height 체인 시작).
- 헤더·사이드바 `shrink-0`. 본문 `flex-1 min-h-0 overflow-y-auto` — **스크롤은 내부 영역에서만**.
- `min-h-screen` 금지, magic number `calc()` 금지.

### 4.6 표
- 행 40px / 헤더 44px / 셀 `px-3`. 텍스트=좌, 숫자=우, 액션=우. zebra 줄무늬 금지. hover `bg-gray-50`.
- 단, 키워드 검색 결과 표의 "구좌 구성" 컬럼은 칩 wrap 위해 **행 높이 가변 허용** (예외).

### 4.7 모달/카드/버튼
- 모달 `max-w-md`(448px) 기본. 카드 `rounded-lg` + 1px border, **shadow 금지**. 버튼 `h-9`.

### 4.8 사이드바 여닫이
- 펼침 `w-60`(240px, 아이콘+라벨) ↔ 접힘 `w-14`(56px, 아이콘만). 모바일(추후) drawer.
- 트리거: 헤더 햄버거 + `Ctrl+B`. 상태 localStorage 저장. `transition-[width] 200ms`.

### 4.9 반응형 — MVP는 데스크톱 전용(≥1024px), 반응형은 Phase 2
미리 준비할 7가지: ①모든 컨테이너 `min-w-0` ②표 부모 `overflow-x-auto` ③고정 px 금지·Tailwind 토큰만 ④사이드바/헤더/모달 컴포넌트화 ⑤`@click`만 사용(hover 의존 금지) ⑥input `min-w-0`+flex-1 ⑦이미지 `max-w-full h-auto`.

### 4.10 상태 4종 (모든 화면 의무)
Loading=`USkeleton` / Empty=`UEmptyState`+CTA / Error=`UAlert`+재시도 / Success=토스트.

### 4.11 강제 메커니즘
Nuxt UI 컴포넌트 우선 · 색상 직접 코드 금지(토큰만) · arbitrary value 최소화 · max-w 제한.

## 6. 페이지 설계

### 6.1 홈 (`/app`)
인사말 없음. 6블록 간략 배치:
1. **이용 현황 바** — 플랜 + 이용 기간 D-day (`users.plan_expires_at`)
2. **키워드 검색 카드** — 빠른 검색 입력창 포함 → `/app/search`
3. **순위 추적 카드** — 이동 버튼만 (준비중 표시 X)
4. **최근 분석한 키워드** — 검색 이력 5건 (`search_history`)
5. **확장 프로그램 카드** — 스마트 브랜딩 서포터 다운로드 (기존 베타 안내·노션 가이드 링크)
6. **문의 카드** — 카카오톡 채널 `http://pf.kakao.com/_mCQGG/chat`

### 6.2 키워드 검색 (`/app/search`) — 핵심 페이지
- **입력**: 페이지 상단 인라인 (모달 X). 키워드 여러 개 줄바꿈 입력.
- **처리**: Worker가 키워드별 ①검색광고 API(검색량·경쟁도) ②`m.search.naver.com` fetch→구좌 마커 파싱 ③24h 캐시. 키워드별 진행 상태 스트리밍.
- **결과 표 (5컬럼)**:

| 컬럼 | 내용 |
|------|------|
| 키워드 | 텍스트 |
| 검색량 (P+M) | `1,200 + 8,400` 병기 (PC+모바일, 합계는 상세에서) |
| **구좌 구성** | **핵심·가장 넓은 컬럼.** SERP 순서대로 칩 나열 |
| 경쟁도 / 유형 | 경쟁도(높음/중간/낮음) + 키워드 유형(1~5, 룰 추후) |
| 상세 | 보기(→) |

- **구좌 구성 칩**: 순서번호(①②③ = SERP 위→아래) + 섹션 표시명(네이버 실제 이름: `건강·의학 인기글`, `관련 사이트`, `함께 많이 찾는`) + 카드 수(`〈블10〉`) + 구좌별 색상. 칩 많으면 행 높이 가변.
- **행 클릭 → 우측 슬라이드 패널**: 구좌 세로 펼침 + 검색량 PC/M 분리 + 연관 검색어 + `순위 추적 추가` 버튼.
- **CSV 내보내기**: 주요 구좌 개수 컬럼 + 구좌 구성 전체 텍스트 + PC/모바일 분리.

### 6.3 순위 추적 (`/app/tracking`) — 이번 세션 보류
- 정의: (타겟 키워드 + 내가 발행한 계정 ID/글) 등록 → 주기적 SERP 순위 측정 → 시계열 기록.
- 예: `청주 한의원` + `abcd19431` 블로그 → 발행 3h후 4위 → 1일후 7위 → 2일후 14위.
- 사용자가 기존 다른 프로그램(uni-portal/checker, SB_CHECKER)에 로직 보유 → **추후 이관 후 상세 설계**.
- 이번 세션은 라우트 자리만 확보.

### 6.4 설정 (`/app/settings`)
계정 정보(구글 이메일·이름, 읽기전용) / 약관·정책 링크 / 로그아웃. 재수집 주기는 추적 기능 이관 시 추가.

### 6.5 이용권 안내 (`/app/plan`)
현재 플랜(무료) 표시 + "유료 준비 중". 유료 전환 시 플랜 비교·결제 버튼 추가.

## 7. 기술 검증 결과 (완료)

기존 사내 코드(`uni-portal/checker`, `SB_CHECKER`) 분석으로 네이버 데이터 수집 방식 검증 완료.

### 7.1 네이버 데이터 수집 3패턴 — 전부 헤드리스 브라우저 불필요

| 패턴 | 방식 | 출처 코드 |
|------|------|----------|
| A | `api.place.naver.com/graphql` JSON API 직접 호출 | `place_rank.py` |
| B | `m.search.naver.com` HTML fetch → 내부 `apiURL`(QRA) 정규식 추출 → 재호출 → JSON | `related_search_checker.py` |
| C | `m.search.naver.com` HTML fetch → HTML 파싱 (`data-block-id` 셀렉터) | `knowledge_checker.py` |

- 모두 순수 HTTP 요청(`requests`). 차단 대비 = 모바일 User-Agent 위장 + `sleep(0.5)` + 재시도 3회.
- **3패턴 전부 Cloudflare Worker에서 실행 가능** (A·B=fetch+JSON, C=fetch+`HTMLRewriter`).

### 7.2 실측 결과
`m.search.naver.com` HTML 1회 fetch(약 873KB)로 한 키워드의 전체 구좌 마커 확인:
- `data-block-id`: 블로그(`review/...blog`), 지식인(`kin/`), 웹(`web/`), 뉴스(`news/`), 인플루언서(`ugc/...influencer`), 파워컨텐츠(`ugc/...powercontents`), 동영상(`video/`), 함께많이찾는(`qra/`)
- 별도 마커: 플레이스(`place-app-root`), 파워링크 광고(`power_link`)
- 두 차례 요청 모두 차단 없이 성공.

### 7.3 결론 + 미검증 1건
- 키워드 검색(구좌 분석) = `m.search.naver.com` HTML 1회 fetch + 마커 파싱으로 구현 가능.
- 순위 추적 = place GraphQL API + m.search HTML 파싱 (기존 코드 JS 포팅).
- **⚠️ 미검증**: Cloudflare Worker는 데이터센터 IP — 네이버가 차단할 가능성. **실배포 후 테스트 필요.** 차단 시 대안(캐시 강화·요청 분산) 존재.

## 8. 데이터 모델 (D1)

이번 세션 확정: `users`, `search_history` 2개. 순위 추적용 2개는 초안(이관 시 최종 확정).
D1 = SQLite 기반. 시각은 모두 TEXT ISO8601. UUID는 앱에서 `crypto.randomUUID()`.

### 8.1 users (확정)
```sql
CREATE TABLE users (
  id              TEXT PRIMARY KEY,
  google_sub      TEXT NOT NULL UNIQUE,             -- 구글 OAuth 고유 ID
  email           TEXT NOT NULL,
  name            TEXT,
  picture         TEXT,                             -- 구글 프로필 이미지
  status          TEXT NOT NULL DEFAULT 'pending',  -- pending|approved|suspended
  plan            TEXT NOT NULL DEFAULT 'free',     -- free|pro
  plan_expires_at TEXT,                             -- null=무기한 (홈 D-day)
  approved_at     TEXT,
  created_at      TEXT NOT NULL,
  last_login_at   TEXT
);
```
- 승인제: 구글 로그인 성공 시 행 생성되나 `status='pending'`. 운영자 승인 시 `approved`. `suspended`면 차단.
- `plan_expires_at` → 홈 이용 기간 D-day.

### 8.2 search_history (확정)
```sql
CREATE TABLE search_history (
  id            TEXT PRIMARY KEY,
  user_id       TEXT NOT NULL REFERENCES users(id),
  keyword       TEXT NOT NULL,
  pc_volume     INTEGER,
  mobile_volume INTEGER,
  competition   TEXT,                  -- 높음|중간|낮음
  sections_json TEXT,                  -- 구좌 구성 결과 (JSON)
  keyword_type  INTEGER,               -- 1~5, nullable (룰 추후)
  created_at    TEXT NOT NULL
);
CREATE INDEX idx_search_history_user ON search_history(user_id, created_at DESC);
```
- 키워드 검색 1건 = 1행. 홈 "최근 분석한 키워드" + 향후 사용량 카운트 겸용.
- `sections_json` 구조:
```json
[
  { "order": 1, "type": "powerlink",    "label": "파워링크",        "count": 10 },
  { "order": 2, "type": "place",        "label": "플레이스",        "count": 8 },
  { "order": 3, "type": "popular_post", "label": "건강·의학 인기글", "cards": { "blog": 10 } },
  { "order": 4, "type": "kin",          "label": "지식인",          "count": 1 },
```

### 8.3 track_targets / rank_measurements (초안 — 순위 추적 이관 시 확정)
```sql
CREATE TABLE track_targets (
  id           TEXT PRIMARY KEY,
  user_id      TEXT NOT NULL REFERENCES users(id),
  keyword      TEXT NOT NULL,           -- 타겟 키워드
  account_id   TEXT NOT NULL,           -- 내 계정 아이디 (blog.naver.com/{account_id})
  label        TEXT,                    -- 표기용 별칭 (선택)
  created_at   TEXT NOT NULL
);

CREATE TABLE rank_measurements (
  id           TEXT PRIMARY KEY,
  target_id    TEXT NOT NULL REFERENCES track_targets(id),
  measured_at  TEXT NOT NULL,           -- 측정 시각
  rank         INTEGER,                 -- 순위 (null=미노출)
  post_title   TEXT,                    -- 측정 시점에 잡힌 글 제목
  post_url     TEXT,                    -- 잡힌 글 링크
  post_id      TEXT                     -- 글번호(logNo) — 글 교체 감지용
);
```
- `rank_measurements`는 측정마다 "그때 잡힌 글"의 제목·링크·글번호를 기록 → 시계열에서 글 교체 시점 추적 가능.

### 8.4 데이터 격리 룰

## 10. 인증 흐름 (Google OAuth + JWT)

### 10.1 전체 흐름
```
[웹앱 /login] "구글로 로그인"
 → GET /api/auth/google → 구글 OAuth 동의 URL로 redirect
 → [구글 로그인·동의]
 → GET /api/auth/google/callback?code=...
    ① code → 구글 토큰 교환 (id_token)
    ② id_token 검증 → { sub, email, name, picture }
    ③ D1 users 조회 (google_sub)
       · 없음 → INSERT (status='pending')
       · 있음 → last_login_at 갱신
    ④ JWT 발급 (status 무관 발급)
    ⑤ 토큰 전달 → 웹앱 /app redirect
 → 웹앱이 GET /api/me로 status 확인
    · approved → 정상 / pending → 승인대기 화면 / suspended → 정지 화면
```

### 10.2 JWT 정책
| 항목 | 값 |
|------|-----|
| 알고리즘 | HS256 (secret은 Worker Secret) |
| 페이로드 | `{ sub: userId, email, iat, exp }` |
| 만료 | 7일 |
| 정지 즉시 반영 | 매 요청 시 `users.status` D1 확인 → suspended면 차단 |

### 10.3 승인제 처리
- 신규 구글 로그인 → `users` 행 생성(`status='pending'`).
- pending도 JWT 발급 → 로그인 상태 유지, 웹앱은 "승인 대기 중" 화면.
- 기능 API(`/api/search` 등)는 `approved`만 통과 (pending → 403).
- 운영자 승인: MVP는 D1 직접 `UPDATE`(`wrangler d1 execute`). 관리자 페이지는 추후.

### 10.4 토큰 저장 + 도메인 — 단계적 방침
```
MVP 개발/베타  : 무료 도메인(pages.dev/workers.dev)
               토큰 = localStorage + Authorization 헤더
               XSS는 CSP 헤더로 완화, authStore 추상화(전환 대비)
유료 전환 시점 : 자체 도메인 확보 → app.도메인 / api.도메인
               httpOnly + Secure + SameSite 쿠키로 전환
```
- 무료 도메인은 웹앱(pages.dev)·Worker(workers.dev)가 서로 다른 도메인이라 httpOnly 쿠키 공유 불가 → MVP는 localStorage 불가피.
- 토큰 저장을 `authStore`로 추상화 → 유료 전환 시 쿠키 방식으로 교체 용이.

## 11. 보안 + 유료 전환 토대

> 사용자가 보안 설계를 위임. 코드 구성 후 사용자가 다른 방식으로 피드백 예정.

### 11.1 보안 항목
| 영역 | 조치 |
|------|------|
| 인증·인가 | JWT HS256 서명 검증·만료 확인 / 매 요청 `users.status` 확인 / `approved`만 기능 API |
| 데이터 격리 | 모든 사용자 쿼리 `WHERE user_id=?` + `requireUser()` 헬퍼 |
| 시크릿 관리 | NAVER 키·Google client_secret·JWT secret → Worker Secret. git·코드 노출 금지 |
| 통신 | HTTPS(Cloudflare 기본) / CORS 화이트리스트 exact 매칭 |
| SQL injection | D1 `.bind()` 파라미터 바인딩 필수, 문자열 조합 금지 |
| XSS | CSP 헤더 / Vue 자동 이스케이프 / localStorage 토큰은 CSP로 보완 |
| 입력 검증 | 키워드 길이(1~50자)·문자 제한 |
| 남용 방지 | Rate limiting (계정·IP당 요청 제한) |
| 봇 차단 | 승인제(1차) + Turnstile(추후 선택) |

### 11.2 유료 전환 토대 (코드 자리만, 구현은 유료 시점)
| 항목 | 지금 (토대) | 추후 (구현) |
|------|------------|------------|
| 플랜 | `users.plan`/`plan_expires_at` 컬럼 존재 | — |
| 사용량 측정 | `search_history` count로 집계 가능 | 월별 한도 집계 |
| 게이팅 포인트 | `/api/search`에 `checkQuota(user)` 함수 위치만 분리, MVP는 항상 통과 | 무료 한도 적용 |
| 결제 | — | `POST /api/plan/*` + 토스페이먼츠/포트원 연동 |
| 약관 | `/terms` `/privacy` 페이지 자리 | 내용 작성 |

핵심: 유료 기능을 나중에 얹을 자리만 미리 잡고 MVP는 전부 통과 → 코드 갈아엎지 않고 유료 전환.
